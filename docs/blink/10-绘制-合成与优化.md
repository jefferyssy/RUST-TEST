# 10 — 绘制合成与优化：BatchOptimizer + 实例化渲染 vs PropertyTrees + PaintArtifact + cc Layer

## 问题/目标

绘制阶段生成 DisplayList 后，渲染器需要将其转换为 GPU 绘制调用。关键优化目标：
- **减少 draw call**：合并相同状态的绘制命令（同色矩形 → 一次实例化 draw）
- **遮挡剔除**：被不透明区域完全覆盖的命令无需绘制
- **变换/透明度免重建**：CSS `transform` / `opacity` 变更不应触发整个 DisplayList 重建
- **滚动优化**：滚动时不需要重建绘制命令，仅平移已有内容

## 我们的实现

```rust
    /// 遮挡剔除：移除被不透明区域完全覆盖的 FillRect
    pub fn occlusion_cull(list: &mut DisplayList) -> BatchStats {
        let commands = std::mem::take(&mut list.commands);
        let mut result = Vec::new();
        let mut opaque_regions: Vec<(f32, f32, f32, f32)> = Vec::new();

        for cmd in commands {
            match &cmd {
                PaintCommand::FillRect { rect, color, .. } => {
                    if color.a == 255 {  // ★ 完全不透明
                        let right = rect.x + rect.width;
                        let bottom = rect.y + rect.height;
                        // 检查是否被已有不透明区域完全覆盖
                        let is_covered = opaque_regions.iter().any(|(ox, oy, or, ob)| {
                            *ox <= rect.x && *oy <= rect.y && *or >= right && *ob >= bottom
                        });
                        if !is_covered {
                            opaque_regions.push((rect.x, rect.y, right, bottom));
                            result.push(cmd);
                        }
                        // else: ★ 被遮挡 → 丢弃
                    } else {
                        result.push(cmd);
                    }
                }
                _ => result.push(cmd),
            }
        }
        list.commands = result;
        stats
    }
}
```

**occlusion_cull 效果**：白色 body 背景覆盖全窗口 → 所有被 body 覆盖的底层 FillRect 被丢弃。

**当前限制**：
- `optimize()` 仅统计，不实际合并（保留全部命令）
- `occlusion_cull()` 仅在渲染时调用，非 DisplayList 构建时
- 不支持跨元素合并（每个元素独立产生命令）

### wgpu 实例化渲染

来源：[crates/renderer/src/wgpu_backend.rs](../../crates/renderer/src/wgpu_backend.rs#L912)

```rust
impl WgpuBackend {
    fn render(&mut self, display_list: &DisplayList) {
        // 1. 分类收集：FillRect→border, Border→rects, BoxShadow→shadow, Text→text
        let mut rect_instances: Vec<f32> = Vec::new();
        let mut border_instances: Vec<f32> = Vec::new();
        let mut shadow_instances: Vec<f32> = Vec::new();
        let mut text_commands: Vec<&PaintCommand> = Vec::new();
        Self::collect_rects_and_text(commands, &mut rect_instances, ...);

        // 2. 写实例缓冲（所有相同类型在一次 write_buffer 中）
        self.queue.write_buffer(&self.instance_buf, 0, bytemuck::cast_slice(&rect_instances));
        self.queue.write_buffer(&self.border_instance_buf, 0, ...);
        self.queue.write_buffer(&self.shadow_instance_buf, 0, ...);

        // 3. 单个 render pass，4 个 draw call（Shadow → Border SDF → Rect → Text）
        let mut pass = encoder.begin_render_pass(...);
        pass.set_pipeline(&self.shadow_pipeline);
        pass.draw(0..6, 0..shadow_count as u32);     // ★ 1 draw call = 所有阴影
        pass.set_pipeline(&self.border_pipeline);
        pass.draw(0..6, 0..border_count as u32);     // ★ 1 draw call = 所有圆角矩形
        pass.set_pipeline(&self.rect_pipeline);
        pass.draw(0..6, 0..rect_count as u32);       // ★ 1 draw call = 所有直角矩形
        pass.set_pipeline(&self.text_pipeline);
        pass.draw(0..6, 0..text_count as u32);       // ★ 1 draw call = 所有文本字形

        self.queue.submit(Some(encoder.finish()));
        output.present();
    }
}
```

**实例化策略**：
- 每种绘制类型 = 1 个 pipeline + 1 个实例缓冲
- 同类型所有元素合并为 1 个 instanced draw call（`draw(0..6, 0..N)`）
- 四边形顶点缓冲在所有 pipeline 间共享（6 顶点 = 2 个三角形）
- 实例数据包含具体位置、颜色、圆角/阴影参数

### 四个渲染管线

| Pipeline | 输入 | 绘制内容 | Shader 类型 |
|----------|------|---------|------------|
| `shadow_pipeline` | 16 f32/instance | BoxShadow（SDF + 高斯模糊衰减） | WGSL SDF |
| `border_pipeline` | 16 f32/instance | FillRect（SDF 圆角）+ Border（SDF 圆角边框） | WGSL SDF |
| `rect_pipeline` | 8 f32/instance | 直角边框四边（4 段窄矩形） | WGSL 简单着色 |
| `text_pipeline` | 12 f32/instance | 文本字形（字形图集纹理采样） | WGSL 纹理 |

```rust
    if c <= 0.04045 { c / 12.92 }
    else { ((c + 0.055) / 1.055).powf(2.4) }
```

### PaintArtifact + PaintChunk

```cpp
class PaintArtifact {
    Vector<DisplayItem> display_items_;    // 扁平数组
    Vector<PaintChunk> chunks_;            // 分块索引 + 属性 ID

    // 提交到合成器
    void CommitToCompositor(LayerListBuilder* builder) {
        for (const PaintChunk& chunk : chunks_) {
            // 每个 PaintChunk → 1 个 cc::Layer
            cc::Layer* layer = builder->CreateLayer();
            layer->SetPropertyTrees(chunk.properties);  // transform/clip/effect
            layer->SetDisplayItems(chunk.begin, chunk.end);
        }
    }
};
```

### cc::Layer 树 → 合成器帧

```cpp
        // 2. tile 化每层的 DisplayItemList

        // 4. 生成 CompositorFrame
        // 5. 提交到 viz 进程
```

### 合成器线程 vs 主线程

```
主线程（Blink）:
  DOM → Style → Layout → Paint → Commit (PaintArtifact → cc::Layer)

合成器线程（Impl）:
  Receive Commit → CalcDrawProps → Raster (tile worker) → Activate → Draw → Submit
                                                                          ↓
GPU 进程 (viz):
  Aggregate CompositorFrames → Render → Display
```

**关键：滚动不阻塞主线程**
- 合成器线程可以独立处理 scroll offset，无需主线程重新 layout/paint
- 仅当滚动区域超出已光栅化的 tile 时，才请求主线程光栅化新 tile

## 优劣势分析

| 维度 | 我们 | Blink |
|------|------|-------|
| **合批方式** | 实例化渲染：同类型 → 1 个 instanced draw | cc::Layer 合并 + SkPicture 录制 + tile 化 |
| **draw call 数** | 固定 4 个（shadow/border/rect/text） | 按 PaintChunk 数量（通常多但 tile 化后可控） |
| **遮挡剔除** | `occlusion_cull()` 简单矩形覆盖检查 | cc::Layer 的 `UpdateDrawProperties` + 可见性计算 |
| **变换/透明度** | 嵌入实例数据，变更需重建实例缓冲 | PropertyTrees 分离，变更仅更新属性树节点 |
| **滚动** | 全量重建（DOM 变更 → relayout） | 合成器线程独立处理（不阻塞主线程） |
| **光栅化** | 单帧全量重绘（所有实例每次渲染） | 分 tile 增量光栅化 + 缓存 |
| **多进程合成** | 无（单进程） | viz 进程聚合多源 CompositorFrame |
| **纹理图集** | `TextureAtlas` 结构体（stub，未用于实际渲染） | Skia 的 GrTextureAtlas + 字形图集 |
| **SRGB 校正** | 手动 `srgb_to_linear` 函数 | Skia 自动处理色彩空间 |

### 我们的优势

1. **draw call 极少**：固定 4 条 instanced draw → GPU 友好
2. **实例化代码清晰**：收集 → 写缓冲 → 单 pass 绘制，逻辑线性
3. **SDF 圆角渲染**：单一 shader 支持填充 + 边框 + 圆角 → 无需额外纹理
4. **阴影高斯模糊**：在 SDF fragment shader 中直接计算 → 无需离屏渲染
5. **遮挡剔除直接**：40 行代码实现基本遮挡检测

### 我们的劣势

1. **无 PropertyTrees**：修改一个元素的 `opacity` 需要重建整个实例缓冲
2. **无 tile 光栅化**：视口外元素仍生成实例数据 → GPU 做不必要的工作
3. **无合成器线程**：所有渲染在主线程同步完成 → 卡顿感明显
4. **无滚动优化**：每次滚动 = 全量重建（layout + paint + draw）
5. **无增量绘制**：每帧重建全部实例数据 → 1000 个不变元素被重复处理
6. **occlusion_cull 过于简单**：仅处理矩形完全覆盖，不处理重叠/半透明

### Blink 的优势

1. **PropertyTrees 避免重建**：transform/opacity 变更只更新属性节点，DisplayItem 全复用
2. **合成器线程**：输入事件和滚动在合成器线程处理，不阻塞主线程
3. **tile 化光栅化**：可见区域分 tile 异步光栅化 → 大页面也能 60fps
4. **SkPicture 缓存**：未变更的 DisplayItem 从 SkPicture 缓存取出，跳过录制
5. **GPU 进程隔离**：渲染崩溃不导致主进程崩溃

### Blink 的劣势

1. **架构复杂度极高**：PropertyTrees + PaintArtifact + cc::Layer + TileManager + viz → 数十万行代码
2. **多进程 IPC 开销**：Commit → Submit → Aggregate 三跳 IPC
3. **tile 管理开销**：TileManager 的内存管理和缓存失效规则复杂

## 改进方向

### 短期

1. **实际合并同色 FillRect**：在 `optimize()` 中合并连续同色矩形为更大的 FillRect
2. **视口剔除**：在收集实例数据时跳过完全在视口外的命令
3. **SRGB 校正批量预计算**：CSS 颜色解析时即转为线性，避免每次渲染转换

### 中期

4. **PropertyTrees 引用**：显示列表不存属性值，而存属性树节点 ID，属性变更只更新树
5. **增量实例缓冲**：仅更新脏命令对应的实例数据，复用其他实例数据
6. **纹理图集实际应用**：用 `TextureAtlas` 打包图标的 Image 命令

### 长期

7. **合成器线程**：将渲染和输入处理移到独立线程，主线程只负责 DOM → Paint 提交
8. **tile 光栅化**：将 DisplayList 分 tile 异步光栅化为 GPU 纹理
9. **多帧流水线**：Frame N 在渲染，Frame N+1 在布局，实现重叠执行
