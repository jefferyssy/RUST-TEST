# 12 — 整合：全管线数据流端到端对照

## 问题/目标

以一个具体的用户交互场景为例，追踪从 **用户点击按钮** 到 **屏幕像素更新** 的完整数据流，对比双方在每个环节的实现差异。

## 场景设定

**用户操作**：在 Todo App 中点击 "Add" 按钮，添加一个新的 todo 项。

**预期效果**：
1. `click` 事件触发 JS 回调
2. 回调修改 DOM（创建新 `<li>` 并 append 到 `<ul>`）
3. 样式重新计算
4. 布局重新计算（新 `<li>` 需要位置和尺寸）
5. DisplayList 重建
6. 屏幕更新

## 我们的全管线

```
┌──────────────────────────────────────────────────────────────────┐
│  winit WindowEvent::MouseInput                                   │
│  → App::handle_click()                                           │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 1 — Hit Test                                               │
│   HitTester::hit_test(&layout_root, x, y)                       │
│   → 深度优先遍历布局树                                           │
│   → 检查每个 LayoutBox.rect 是否包含 (x, y)                      │
│   → 返回最深的匹配节点（含 DOM node 引用）                       │
│   → 若为 Text 节点，向上查找父 Element                           │
│                                                                  │
│ 复杂度: O(布局树节点数)                                          │
│ 输出: Rc<RefCell<Node>> (目标 DOM 节点)                          │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 2 — 事件派发                                               │
│   EventDispatcher::dispatch(&target_node, &mouse_event)          │
│   → build_path(target): [target, parent, ..., document_element]  │
│   → 捕获阶段 (自上而下, capture=true 监听器)                      │
│   → 目标阶段 (所有监听器)                                         │
│   → 冒泡阶段 (自下而上, capture=false 监听器)                     │
│                                                                  │
│   回调示例:                                                      │
│     function() {                                                 │
│       const li = document.createElement('li');                   │
│       li.textContent = input.value;                              │
│       list.appendChild(li);                                      │
│     }                                                            │
│                                                                  │
│   内部操作:                                                      │
│     Node::create_element("li")  → Rc::new_cyclic(...)            │
│     Node::set_text_content("新todo")  → 创建 Text 子节点          │
│     Node::append_child(li)                                       │
│       → 设置 parent/prev_sibling/next_sibling                    │
│       → children.push(li)                                        │
│       → mark_dirty(true)                                         │
│                                                                  │
│ 复杂度: O(祖先深度) + O(监听器数)                                │
│ 输出: DOM 树已修改 + dirty 标记已设置                            │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 3 — 样式重算 (relayout → compute_dom_styles)               │
│   for each node in DOM (depth-first):                             │
│     NodeType::Element →                                          │
│       1. 解析 inline style 属性                                   │
│       2. match_selectors(element, &stylesheets)                  │
│          → ★ 三重循环: sheets × rules × selectors               │
│          → 每个选择器对每个元素做 tag/class/id 匹配              │
│       3. compute_element_style(...)                              │
│          → 按特异性排序 (低→高)                                   │
│          → 依次 apply: 后覆盖前                                   │
│          → 继承父元素的可继承属性                                 │
│          → 应用 UA 默认样式                                       │
│       → styles.insert(ptr, computed_style)                       │
│                                                                  │
│ 复杂度: O(Elements × Rules × SelectorParts)                      │
│ 输出: HashMap<RcPtr, ComputedStyle> (全量新映射)                 │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 4 — 布局树重建 (build_layout_tree)                         │
│   build_layout_box(dom_root, &styles, 0, text_measurer)          │
│   for each DOM node (depth-first):                                │
│     1. 查 computed_style → 确定 BoxType:                          │
│        display:flex → FlexContainer                              │
│        <span>/<a> 默认 → Inline                                  │
│        <div>/<li> 默认 → Block                                   │
│        position:absolute → Absolute                              │
│     2. 创建 LayoutBox(box_type, dom_node)                        │
│     3. apply_shorthand_property(style, &mut box):                │
│        → 解析 padding/margin/border 到 EdgeSizes                 │
│     4. 递归构建子节点 → box.append_child(child_box)              │
│     5. 估算容器尺寸 = max(子节点宽度) + padding                  │
│                                                                  │
│ 复杂度: O(Elements) 全量重建                                     │
│ 输出: 全新的 LayoutBox 树 (所有权转移)                           │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 5 — 布局计算 (LayoutEngine::layout)                        │
│   root.rect.width = viewport.width                                │
│   calculate_sizes(root, viewport, None):                          │
│     match node.box_type:                                          │
│       FlexContainer → FlexLayout.layout(&mut taffy, ...)         │
│         → taffy tree 构建 → compute_layout → 写回 rect           │
│       Block → BlockLayout.layout(node, viewport)                 │
│         → 逐子节点 Y 堆叠                                         │
│         → 外边距折叠                                              │
│         → text-align 对齐                                        │
│       GridContainer → GridLayout.layout(...)                     │
│       Absolute/Fixed → BlockLayout + 偏移                        │
│     calculate_sizes_children → 递归子节点                         │
│     shrink_to_content → 收缩包裹                                 │
│     子节点布局后重跑 BlockLayout (修正兄弟节点漂移)               │
│   root.rect.height = max(root.rect.height, viewport.height)       │
│   PositionedLayout.layout(root, viewport) → 绝对/相对/固定偏移   │
│                                                                  │
│ 复杂度: O(LayoutNodes) 全量重排                                  │
│ 输出: 每个 LayoutBox.rect 已填充最终位置                          │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 6 — DisplayList 构建 (DisplayListBuilder::build)           │
│   process_node(layout_root):                                      │
│     for each node:                                                │
│       0. extract_box_shadow → BoxShadow cmd                      │
│       1. extract_bg_color → FillRect cmd                         │
│       2. extract_border → Border cmd                             │
│       3. extract_text → Text cmd (含 placeholder 支持)            │
│       4. 递归子节点                                               │
│   sort_by_z_order(): FillRect(0) → Border(2) → Text(3) → ...     │
│                                                                  │
│ 复杂度: O(LayoutNodes) 全量重建                                  │
│ 输出: DisplayList { commands: Vec<PaintCommand> }               │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 7 — GPU 渲染 (WgpuBackend::render)                         │
│   1. collect_rects_and_text(commands):                            │
│      FillRect → push_border_instance (SDF 圆角)                  │
│      Border → push_rect (4 边分别) 或 push_border_instance       │
│      BoxShadow → push_shadow_instance                            │
│      Text → text_commands 列表                                   │
│                                                                  │
│   2. prepare_text → 字形光栅化到图集+生成实例数据                 │
│   3. 写实例缓冲 + 屏幕 uniform 到 GPU                             │
│   4. surface.get_current_texture()                                │
│   5. begin_render_pass (Clear WHITE)                              │
│   6. Shadow pipeline → draw(0..6, 0..shadow_count)               │
│   7. Border pipeline → draw(0..6, 0..border_count)               │
│   8. Rect pipeline → draw(0..6, 0..rect_count)                   │
│   9. Text pipeline → draw(0..6, 0..text_count)                   │
│   10. submit + present                                           │
│                                                                  │
│ 复杂度: O(PaintCommands) 全量                                    │
│ 输出: 屏幕像素更新                                               │
└──────────────────────────────────────────────────────────────────┘
```

**总耗时估算** (500 节点页面，1 个 DOM 变更)：
- HitTest: ~10μs (DFS 500 节点)
- Event dispatch: ~10μs
- Style recalc: ~200μs (500 元素 × 20 规则)
- Layout tree build: ~100μs (500 节点)
- Layout compute: ~1000μs (taffy + block)
- DisplayList build: ~100μs
- GPU render: ~500μs (实例数据上传 + draw)
- **总计: ~2ms** (对于 500 节点 demo，FPS 维持 60 可行)

**瓶颈预测**：节点数增至 5000 时，样式重算 → O(5000×20) = 100K 匹配 ≈ 5ms，布局 ≈ 5ms，总计 ~12ms（仍可达 60fps）。

## Blink 的全管线

```
┌──────────────────────────────────────────────────────────────────┐
│ 浏览器进程 → IPC → 渲染进程                                      │
│   InputHandler::HandleMouseEvent(mouse_data)                     │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 1 — HitTest (cc::LayerTreeHostImpl, 合成器线程)             │
│   LayerTreeHostImpl::FindLayerThatIsHitByPointInTouchAction()    │
│   → 利用 PropertyTrees 中的变换矩阵                               │
│   → 裁剪树跳过不可见区域                                          │
│   → 返回被点击的 cc::Layer + 坐标                                 │
│                                                                  │
│ 如果无 JS touch/wheel 阻止 → 合成器线程直接处理（无需主线程）     │
│ ★ 关键：被动事件在此直接处理，无主线程等待                       │
└────────────────────────┬─────────────────────────────────────────┘
                         │ (IPC 到主线程，仅当有 non-passive 监听器)
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 2 — Event Dispatch (主线程)                                 │
│   EventHandler::HandleMouseEvent(event)                           │
│   → HitTest (blink 侧，确认 DOM 节点)                             │
│   → EventDispatcher::Dispatch(node, event)                        │
│     → EventPath::BuildPath (含 Shadow DOM、slot 分配)            │
│     → 捕获 → 目标 → 冒泡 (三阶段)                                │
│                                                                  │
│   JS 回调操作:                                                    │
│     document.createElement('li')                                  │
│       → MakeGarbageCollected<HTMLLIElement>(document)             │
│     li.textContent = "新todo"                                    │
│       → Node::setTextContent → 创建 Text 子节点                  │
│       → ChildrenChanged() → SetNeedsStyleRecalc() (仅新 li)     │
│     list.appendChild(li)                                         │
│       → ContainerNode::AppendChild → InsertBefore                │
│       → ChildrenChanged()                                        │
│       → SetNeedsStyleRecalc() (向上传播至 list 祖先)            │
│       → ScheduleLayoutTreeUpdate()  ← ★ 异步调度帧               │
│                                                                  │
│ 输出: DOM 已修改 + 脏标记已沿子树传播                             │
└────────────────────────┬─────────────────────────────────────────┘
                         │ (异步: 等待 BeginMainFrame)
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 3 — Style Recalc (主线程, BeginMainFrame)                   │
│   StyleResolver::UpdateStyleAndLayoutTree()                       │
│   for (Element* dirty : dirty_elements_)  // ★ 仅脏元素          │
│     1. StyleResolver::ResolveStyle(element, parent_style)         │
│        → CascadePriority 排序 (6 级: important/origin/scope/...) │
│        → StyleCascade::Apply (分组 CoW ComputedStyle)             │
│        → ★ 大部分属性组与父级共享 (Arc::make_mut 仅修改变更组)   │
│     2. StyleDifference diff = ComputeDifference(old, new)         │
│        → diff == kNoChange?                                       │
│           ✓ → 子节点不需要传播样式 → 跳过子树                     │
│           → diff.NeedsLayout()? → 标记 LayoutObject 脏            │
│           → diff.NeedsInheritedUpdate()? → 继续传播到子节点        │
│     3. 若 LayoutObject 不存在且需要 → CreateLayoutObject(...)    │
│        → 按 display 值创建对应子类 (LayoutBlockFlow / etc.)      │
│                                                                  │
│ 复杂度: O(脏元素数 + 受影响子树)，通常 << O(总元素数)           │
│ 输出: 仅脏元素 ComputedStyle 更新，脏树的 LayoutObject 就绪      │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 4 — Layout Update (主线程)                                  │
│   LocalFrameView::UpdateLayout()                                  │
│   root->UpdateLayout()                                            │
│     → 仅 NeedsLayout() 的子树执行 Layout() 虚函数                 │
│       LayoutBlockFlow::UpdateLayout():                            │
│         1. 构建 NGConstraintSpace                                 │
│         2. 检查 Fragment 缓存 (节点+约束 → 片段)                  │
│            ✓ 命中 → ApplyFragment → 跳过 Layout()                │
│            ✗ 未命中 → NGBlockLayoutAlgorithm::Layout()            │
│               → 遍历子节点 → 构建子约束空间 → 子节点.Layout()    │
│               → BFC 调整 (margin collapse / float / clear)       │
│               → 生成不可变 NGPhysicalBoxFragment                  │
│         3. ApplyFragmentToLayoutObject → 更新 rect 等             │
│   ★ 兄弟子树若无脏标记 + 约束空间未变 → Fragment 缓存命中 → 跳过  │
│                                                                  │
│ 复杂度: O(脏子树大小)，其余子树缓存命中                           │
│ 输出: 脏 LayoutObject 的 FrameRect 已更新                         │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 5 — PrePaint (主线程)                                       │
│   PrePaintTreeWalk::Walk(root)                                    │
│   1. 构建 PropertyTrees:                                          │
│      → TransformTree (每个 LayoutObject 的变换节点)               │
│      → ClipTree (overflow: hidden / clip-path 节点)              │
│      → EffectTree (opacity / filter 节点)                         │
│      → ScrollTree (滚动偏移节点)                                  │
│   2. 仅更新变更的属性树分支 (已缓存节点跳过)                       │
│                                                                  │
│ 复杂度: O(脏属性树分支)                                          │
│ 输出: PropertyTrees (transform / clip / effect / scroll 最新值)  │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 6 — Paint (主线程)                                          │
│   PaintController::Paint(root)                                    │
│   1. 遍历 LayoutObject 树                                          │
│   2. 每个 LayoutObject → BoxPainter::Paint(info)                  │
│      → PaintBackground → PaintBoxShadow → PaintBorder →           │
│        PaintForeground (递归子 LayoutObject)                      │
│   3. 生成 DisplayItem:                                            │
│      DisplayItem::EqualsForInvalidation(prev_item)                │
│        ✓ 相同 → 从 SkPicture 缓存取出 (跳过绘制)                  │
│        ✗ 不同 → 录制新 SkPicture + 插入 DisplayItemList           │
│   4. 合并为 PaintChunk (连续同 PropertyTree 状态)                 │
│                                                                  │
│ 复杂度: O(脏绘制区域节点)                                        │
│ 输出: PaintArtifact (DisplayItems + PaintChunks + PropertyTrees) │
└────────────────────────┬─────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 7 — Commit (主线程 → 合成器线程)                            │
│   PaintArtifact::CommitToCompositor(LayerListBuilder)             │
│   1. 按 PaintChunk 创建 cc::Layer                                  │
│   2. 每个 Layer 包含:                                             │
│      → PropertyTreeState (transform/clip/effect node IDs)        │
│      → DisplayItemList (begin_index..end_index)                   │
│   3. Layer 树提交到合成器线程                                     │
│   4. 合成器线程接收 → CompositeToOutputSpace                     │
│   5. LayerTreeHostImpl::CalcDrawProps                             │
│      → 遍历层树计算可见性、绘制区域                               │
│      → 更新 tile 优先级                                            │
│   6. TileManager::UpdateTiles                                     │
│      → 异步光栅化脏 tile (tile worker pool)                       │
│   7. Generate CompositorFrame                                     │
│   8. Submit CompositorFrame → viz 进程                            │
│                                                                  │
│ 复杂度: O(脏层数) + O(脏 tile 数)                                │
└────────────────────────┬─────────────────────────────────────────┘
                         │ (IPC)
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│ Phase 8 — Display (viz 进程)                                      │
│   1. SurfaceAggregator::Aggregate(root_surface_id)                │
│      → 合并多个 Renderer 的 CompositorFrame                       │
│   2. OverlayProcessor::ProcessForOverlays                         │
│      → 硬件叠加层优化 (免合成)                                    │
│   3. DirectRenderer::DrawRenderPass                               │
│      → 使用 Skia / ANGLE 绘制到屏幕                               │
│   4. SwapBuffers → Display                                       │
│                                                                  │
│ 复杂度: O(帧复杂度)                                              │
│ 输出: 屏幕像素更新                                               │
└──────────────────────────────────────────────────────────────────┘
```

## 关键时序对比

| 时间点 | 我们 | Blink |
|--------|------|-------|
| DOM 变更后 | 同步立即执行 Style+Layout+Paint+Render | `ScheduleLayoutTreeUpdate()` 异步调度 |
| 到样式更新 | 立即 (0ms) | 等待 BeginMainFrame (~8ms 内) |
| 到帧完成 | 立即 (在 handle_click 中) | 下一帧 vsync (累计 ~16ms) |
| 总延迟 (用户→屏幕) | ~2ms | ~16ms (但在帧速率方面更平滑) |
| 中断能力 | 无 (同步阻塞) | 有 (被动事件由合成器线程处理) |

## 整体差距总结

| 层级 | 我们的现状 | Blink 的实现 | 核心差距 |
|------|-----------|-------------|---------|
| DOM 内存 | Rc+RefCell 引用计数 | Oilpan GC | GC 惰性分配 + 循环安全 |
| 事件存储 | 每元素 HashMap (48B) | 惰性 EventTargetData | 空元素零开销 |
| 事件传播 | 三阶段传播 | 三阶段 + ShadowDOM + 被动优化 | ShadowDOM + 合成器快速路径 |
| 选择器匹配 | O(E×R×S) 全表遍历 | BloomFilter + RTL 匹配 | 快速拒绝 60-80% 规则 |
| 样式计算 | HashMap 全量克隆 | 分组 CoW ComputedStyle | 共享引用 + 按需复制 |
| 样式更新 | 全量重建 ComputedStyle | StyleRecalcChange 增量 | 仅脏元素重算 |
| 布局树 | enum + Vec 所有权 | LayoutObject 继承 + GC | 增量构建 + 虚函数 |
| 布局计算 | 手写分发 + taffy | LayoutNG 算法族 | 完整 BFC + Fragment 缓存 |
| 布局更新 | 全量 relayout | NeedsLayout 脏标记 + 缓存 | 仅脏子树重排 |
| DisplayList | 扁平 enum + 全局排序 | DisplayItem + PaintChunk + PropertyTrees | 属性分离 + 增量比较 |
| 绘制优化 | 实例化 4 draw call | SkPicture 缓存 + tile 光栅化 | 细粒度缓存 + 异步光栅化 |
| GPU 渲染 | wgpu + WGSL 自研 shader | Skia + cc + viz | 完整 2D 图形生态 |
| 合成 | 实例化单 pass | 合成器线程 + 多进程 | 异步 + 安全隔离 + 滚动 |
| 数据流 | 同步全量重建 | 异步增量 + 自适应 | 批处理 + 调度 + 中断 |
