# 08 — 布局增量更新：全量重建 vs 脏标记 + 局部重排

## 问题/目标

浏览器中 DOM 变更频繁发生（JS 修改 DOM、用户输入、CSS 动画），每次变更都需要更新屏幕。关键约束：
- 页面可能有数千个 DOM 节点，但每次变更通常只影响少数几个
- 布局计算是 O(N) 操作，全量重建在 10000 节点时可能达到 100ms+（超出 16ms 帧预算）
- 需要尽可能复用上一次的布局结果，只重新计算受影响的部分

核心思路：**脏标记传播 + 增量更新**——标记哪些节点需要重新计算，仅更新脏子树。

## 我们的实现

### 当前状态：全量重建

来源：[crates/renderer/src/window.rs](../../crates/renderer/src/window.rs#L418)

```rust
impl App {
    /// 重建布局和 DisplayList（窗口 resize 或 DOM 变更后调用）
    fn relayout(&mut self) {
        // 1. 重新计算样式 —— 遍历整棵 DOM 树
        let mut styles: HashMap<usize, ComputedStyle> = HashMap::new();
        compute_dom_styles(&dom_root, &[], None, &mut styles);

        // 2. 重建布局树 —— 全量 rebuild
        let mut new_root = build_layout_tree(&body, &styles, ...);

        // 3. 执行布局 —— 所有节点重新计算
        self.layout_engine.layout(&mut new_root, viewport);

        // 4. 重建 DisplayList —— 所有绘制命令重建
        let mut builder = DisplayListBuilder::new();
        let new_dl = builder.build(&new_root);

        self.styles = styles;
        self.layout_root = Some(new_root);
        self.display_list = Some(new_dl);
    }
}
```

**触发时机**：
- 窗口 resize → `relayout()`（`WindowEvent::Resized`）
- DOM 变更（点击、键盘输入）→ `relayout()`
- 每次都是**全量重建**整个渲染管线

impl LayoutBox {
    // Phase 3: 脏标记（增量布局）
    pub dirty: bool,

    /// 检查自身或任意后代是否为脏
    pub fn has_dirty(&self) -> bool {
        self.dirty || self.children.iter().any(|c| c.has_dirty())
    }

    pub fn mark_dirty(&mut self) { self.dirty = true; }

    pub fn clear_dirty(&mut self) {
        self.dirty = false;
        for child in &mut self.children {
            child.clear_dirty();
        }
    }

    /// 查找包含指定 DOM 节点的布局框
    pub fn find_layout_node(&self, dom_node: &Rc<RefCell<Node>>) -> Option<&LayoutBox> {
        // 通过 Rc 指针地址比较进行查找
        if let Some(ref node) = self.node {
            if Rc::as_ptr(node) as usize == Rc::as_ptr(dom_node) as usize {
                return Some(self);
            }
        }
        for child in &self.children { ... }
    }
}

### 已有的增量接口（stub）

来源：[crates/layout/src/lib.rs](../../crates/layout/src/lib.rs#L342)

```rust
impl LayoutEngine {
    /// Phase 1: 局部重排（stub —— 当前实际执行全量 layout）
    pub fn partial_layout(&mut self, root: &mut LayoutBox,
                          dirty_nodes: &[Rc<RefCell<Node>>], viewport: Size<f32>) {
        for dirty in dirty_nodes {
            if let Some(layout_node) = root.find_layout_node(dirty) {
                let idx = layout_node as *const LayoutBox as usize;
                self.relayout_node(root, idx, viewport, None);
            }
        }
    }

    fn relayout_node(&mut self, root: &mut LayoutBox, target_ptr: usize,
                     viewport: Size<f32>, parent_box_type: Option<BoxType>) {
        if (root as *const LayoutBox as usize) == target_ptr {
            self.calculate_sizes(root, viewport, parent_box_type);  // ★ 仅重算目标节点
            return;
        }
        for child in &mut root.children {
            self.relayout_node(child, target_ptr, viewport, Some(root.box_type.clone()));
        }
    }

    /// 增量更新：仅布局脏子树
    pub fn update_layout_tree(&mut self, root: &mut LayoutBox, viewport: Size<f32>) {
        if root.has_dirty() {
            self.layout(root, viewport);    // ★ 任何脏节点 → 全树布局
            root.clear_dirty();
        }
    }
}
```

**问题**：`update_layout_tree` 检测到任何脏节点就直接 `self.layout(root)` 全量布局，`clear_dirty()` 也只是清除标记但不做增量优化。脏标记已记录但**不被实际用于跳过非脏子树**。

### 当前数据流：全量重建

```
DOM 变更 (append_child / set_text_content / set_attribute)
  → Node.mark_dirty(true)            ← 脏标记已写，但不被消费
  → relayout()
    → compute_dom_styles()           ← 遍历全 DOM 树
    → build_layout_tree()            ← 全量重建布局树
    → LayoutEngine::layout()         ← 全量重排
    → DisplayListBuilder::build()    ← 全量重建绘制列表
    → WgpuBackend::render()          ← 全量绘制
```

## Blink 的实现

### 脏标记生命周期

```cpp
// ===== 1. 标记脏 =====
void LayoutObject::SetNeedsLayout(
    LayoutInvalidationReason reason,
    MarkingBehavior behavior = kMarkContainerChain) {

    // 向上传播脏标记到所有祖先
    for (LayoutObject* ancestor = this; ancestor; ancestor = ancestor->Parent()) {
        if (ancestor->NeedsLayout())
            break;                           // ★ 遇到已脏祖先，停止传播
        ancestor->SetNeedsLayoutBit(true);   // 设置标记
    }

    // 调度帧更新
    if (Document* doc = GetDocument())
        doc->ScheduleLayoutTreeUpdate();     // ★ 异步调度，非同步执行
}

// ===== 2. 增量更新 =====
void LocalFrameView::UpdateLayout() {
    Lifecycle().AdvanceTo(kLayoutClean);

    // 从根开始，但只处理脏子树
    if (LayoutObject* root = GetLayoutView()) {
        root->UpdateLayout();                // ★ 多态虚函数
    }
}

// ===== 3. LayoutBlockFlow 的增量 UpdateLayout =====
void LayoutBlockFlow::UpdateLayout() {
    if (!NeedsLayout())
        return;                              // ★ 非脏 → 立即返回

    // 1. 构建 NGConstraintSpace
    NGConstraintSpace space = BuildConstraintSpace(this);

    // 2. 检查 Fragment 缓存
    if (auto* cached = LayoutCache().Find(this, space)) {
        ApplyFragment(cached);               // ★ 缓存命中 → 跳过 Layout()
        ClearNeedsLayout();
        return;
    }

    // 3. 执行 LayoutNG 算法
    scoped_refptr<const NGLayoutResult> result =
        NGBlockLayoutAlgorithm(this, space).Layout();

    // 4. 更新几何信息
    ApplyFragmentToLayoutObject(result->PhysicalFragment());

    ClearNeedsLayout();
}
```

### StyleRecalcChange：样式增量传播

```cpp
// 样式引擎的增量更新更精细
enum class StyleRecalcChange {
    kNoChange,            // 样式完全未变
    kInherit,             // 仅继承属性可能改变（传播到子节点）
    kUpdatePseudoElements,// 伪元素需要更新
    kForce,               // 强制重新级联
};

StyleRecalcChange StyleResolver::ResolveStyle(
    Element* element, const ComputedStyle* parent_style) {

    if (!element->NeedsStyleRecalc())
        return StyleRecalcChange::kNoChange;  // ★ 非脏 → 跳过

    // 重新级联
    const ComputedStyle* new_style = CascadeStyle(element, parent_style);

    // 比较新旧样式，决定是否传播到子节点
    StyleDifference diff = ComputeDifference(element->GetComputedStyle(), new_style);
    element->SetComputedStyle(new_style);

    if (diff.NeedsRecompositeLayer())
        element->SetNeedsCompositingUpdate();

    // 返回变更级别
    return diff.NeedsLayout()
        ? StyleRecalcChange::kForce
        : diff.NeedsInheritedUpdate()
            ? StyleRecalcChange::kInherit   // ★ 如 color 变更 → 子节点需更新继承
            : StyleRecalcChange::kNoChange;
}
```

### 增量更新的三个层次

```
Level 1 — 样式增量：
  仅脏元素重新级联 → StyleDifference 决定子节点是否传播
  "修改 .todo-item 的 color" → 仅遍历 .todo-item 子树的继承属性

Level 2 — 布局增量：
  仅 NeedsLayout() 的子树执行 Layout()
  "修改 <div> 的 width" → 仅该 <div> 及其兄弟节点重排

Level 3 — 绘制增量：
  仅重绘 dirty rect 区域，合成器线程异步光栅化
  "修改一个按钮的颜色" → 仅重绘该按钮的 tile
```

### 调度机制：异步批量处理

```cpp
// Blink 不会在每次 DOM 变更后立即渲染，而是通过帧调度器批量处理
void Document::ScheduleLayoutTreeUpdate() {
    if (pending_layout_update_)
        return;                              // ★ 去重：多次变更只调度一次

    pending_layout_update_ = true;
    GetPage()->Animator().ScheduleFrame();   // ★ 请求 BeginMainFrame
}

// 帧处理时批量执行
void LocalFrameView::BeginMainFrame() {
    // 一次处理所有累积的脏标记
    UpdateStyle();                           // 批量样式更新
    UpdateLayout();                          // 批量布局更新
    RunPrePaintLifecyclePhase();
    RunPaintLifecyclePhase();
    CommitToCompositorThread();
}
```

## 优劣势分析

| 维度 | 我们 | Blink |
|------|------|-------|
| **更新粒度** | 全量重建（O(N) 每次） | 增量更新（O(脏子树大小)） |
| **脏标记** | 已定义 `dirty: bool` 但基本未使用 | 三位标记体系（样式/布局/绘制独立） |
| **调度** | 同步：DOM 变更→立即全量重建 | 异步：ScheduleFrame → BeginMainFrame |
| **批处理** | 无：每个变更单独触发 | 有：多个变更合并到一个帧 |
| **样式比较** | 无：每次全量重新级联 | StyleDifference 精确判断传播范围 |
| **Fragment 缓存** | 无 | (节点, 约束空间) → Fragment 缓存 |
| **布局缓存** | 无 | LayoutBox 可标记 `cached_layout_result_` |
| **代码复杂度** | 极简（直接重建） | 高（三级脏标记 + 调度 + 缓存） |

### 我们的优势

1. **代码简单**：无需脏标记传播逻辑、无需缓存失效、无需批处理调度
2. **正确性保证**：每次都是全量重建，不存在增量更新中常见的"漏更"bug
3. **内存精简**：不需要存布局缓存、Fragment 快照
4. **调试容易**：结果完全由当前状态决定，无历史状态干扰

### 我们的劣势

1. **性能瓶颈（最关键）**：1000 个节点、每帧全量重建 → ~50-100ms（超出 16ms 帧预算 3-6 倍）
2. **无样式差异检测**：修改一个 `color` 也触发全量布局重建
3. **无绘制缓存**：修改一个按钮的颜色也重建整个 DisplayList
4. **同步阻塞**：DOM 变更同步执行布局，滚动/动画无机会插入
5. **布局树必须重建**：因为 `build_layout_tree` 是递归构建 `LayoutBox` 并转移所有权，无法原地更新

### Blink 的优势

1. **性能**：通常仅 O(脏子树) 耗时，脏节点数通常远小于总节点数
2. **三级独立**：样式变更不一定触发布局，布局变更不一定触发全量重绘
3. **异步调度**：多次 DOM 变更合并到一个帧，减少重复工作
4. **缓存复用**：约束空间不变的子树直接复用 Fragment

### Blink 的劣势

1. **复杂度极高**：正确维护脏标记 + 缓存失效规则几乎需要整个团队持续投入
2. **缓存失效 bug**：若某个场景未正确标记脏 → "修改了 CSS 但没生效" 类 bug 极难排查
3. **内存开销**：每个 LayoutBox 有 `cached_layout_result_` / `layout_overflow_` 等额外字段

## 改进方向

### 短期

1. **利用现有 dirty 标记跳过样式重算**：`compute_dom_styles` 中检查 `dirty`，非脏节点 + 父级样式未变 → 复用上次 `ComputedStyle`
2. **利用现有 dirty 标记跳过布局树重建**：`build_layout_box` 中检查 `dirty`，非脏节点复用上次 `LayoutBox.rect`
3. **简单位图脏矩形**：在 DisplayList 层记录变更区域，仅重绘该区域

### 中期

4. **布局树原地更新**：改为可修改的布局树（而非全量重建），利用 `partial_layout` 接口
   ```rust
   fn smart_relayout(&mut self) {
       let dirty_dom: Vec<_> = collect_dirty_nodes(&self.document);
       if dirty_dom.len() < total_nodes / 2 {
           self.layout_engine.partial_layout(&mut root, &dirty_dom, viewport);
       } else {
           self.layout_engine.layout(&mut root, viewport);  // 脏节点太多，全量更便宜
       }
   }
   ```
5. **异步调度**：`requestAnimationFrame` 风格，DOM 变更后不立即重排，而是在帧末尾批量处理

### 长期

6. **样式差异检测**：比较新旧 `ComputedStyle`，返回 `StyleDifference`（`NoChange / InheritOnly / NeedsLayout / NeedsPaint`）
7. **Fragment 缓存**：参考 LayoutNG 为子树缓存布局结果，约束不变直接复用
8. **三级独立脏标记**：`style_dirty / layout_dirty / paint_dirty` 分别控制
