# 01 — 总览：架构对比

## 问题/目标

浏览器引擎的核心任务是：将 HTML+CSS+JS 输入转换为屏幕上的像素输出。这一转换过程在业界的标准架构是 **渲染管线**（Rendering Pipeline），由多个独立的阶段串联组成。

本文对照我们的 Rust 实现与 Blink 的 C++ 实现的整体架构，建立后续深入分析的全局框架。

## 我们的实现

### 模块映射

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ toolchain │  │   net    │  │ storage  │   ← 编译/网络/存储（独立工具）
└──────────┘  └──────────┘  └──────────┘
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│   dom    │→│  style   │→│  layout  │→│rend_tree│→│ renderer │
└──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
   W3C DOM      CSS 引擎     布局引擎     DisplayList   wgpu 渲染
```

**依赖方向**：严格单向，`dom ← style ← layout ← render_tree ← renderer`

### 渲染主循环（来自 `crates/renderer/src/window.rs`）

```rust
// WebWindow::run() 内部的渲染循环
fn render(&mut self, backend: &mut WgpuBackend) {
    // 1. 递归计算每个 DOM 节点的 ComputedStyle
    let mut styles = HashMap::new();
    compute_dom_styles(&self.doc.borrow().document_element().unwrap(), 
                       &[], None, &mut styles);

    // 2. DOM + ComputedStyle → LayoutBox 树
    let mut layout_root = build_layout_tree(&root_node, &styles);

    // 3. 计算每个 LayoutBox 的位置和尺寸
    let viewport = Size::new(self.width as f32, self.height as f32);
    self.layout_engine.layout(&mut layout_root, viewport);

    // 4. LayoutBox 树 → PaintCommand 列表
    let mut builder = DisplayListBuilder::new();
    let display_list = builder.build(&layout_root);

    // 5. PaintCommand → GPU 绘制调用
    backend.render(&display_list);
}
```

**特点**：五个阶段在主线程上**同步顺序**执行，每次 DOM 变更触发**全量重建**。

### 编译模式（来自 `crates/toolchain/src/lib.rs`）

```rust
// toolchain 将 HTML+CSS+JS 编译为 Rust 源代码
pub fn compile_body(input_dir: &Path, options: &CompileOptions) -> String {
    // 1. 解析 HTML → HtmlElement 树
    // 2. 解析 CSS → CssRule 列表 → 匹配到元素 → set_style() 调用
    // 3. 解析 JS → 事件处理器模式 → add_event_listener() 调用
    // 4. 输出生成的 Rust 源代码
}
```

这与 Blink 有根本性差异：我们的 JS 在**编译时**翻译为 Rust，而非运行时解释/JIT 编译。

## Blink 的实现

### 模块映射

```
Blink 进程内结构（渲染进程 / Renderer Process）：

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│     DOM      │  │  StyleEngine │  │ LayoutObject │
│ core/dom/    │  │ core/css/    │  │ core/layout/ │
│ (Node,       │  │ (StyleRes-   │  │ (LayoutBox,  │
│  Element)    │→│  olver,      │→│  LayoutNG)   │
│              │  │  Cascade)   │  │              │
└──────────────┘  └──────────────┘  └──────────────┘
                                              │
                                              ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│     viz      │  │     cc       │  │    Paint     │
│  (GPU 进程)  │← │  (合成器)    │← │ platform/    │
│              │  │              │  │ graphics/    │
│              │  │              │  │ paint/       │
└──────────────┘  └──────────────┘  └──────────────┘

跨进线程模型：
  主线程：     DOM → Style → Layout → PrePaint → Paint → Commit
  合成器线程：   Receive Commit → Raster → Activate → Draw → Submit
  Viz 进程：    Aggregate → Render → Display
```

### 渲染主循环（简化）

```cpp
// LocalFrameView::UpdateLifecyclePhases() — 驱动全管线
void LocalFrameView::UpdateLifecyclePhases(DocumentLifecycle::LifecycleState target) {
    // 仅推进到需要的阶段，中间阶段可能被跳过
    if (target >= kStyleClean)
        GetFrame().GetDocument()->UpdateStyleAndLayoutTree();  // Style + Layout

    if (target >= kPrePaintClean)
        RunPrePaintLifecyclePhase();  // 构建 PropertyTrees

    if (target >= kPaintClean)
        RunPaintLifecyclePhase();     // 生成 PaintArtifact

    if (target >= kCompositingInputsClean)
        CommitToCompositorThread();   // 提交到合成器线程
}
```

**关键差异**：
- 每个阶段有独立的"脏标记"检查，未变更的阶段直接跳过
- Paint 之后 Commit 到**合成器线程**异步执行光栅化和显示
- viz 进程独立，处理多进程合成的安全隔离

## 数据流对比

### 我们的数据流

```
winit 事件 → DOM 修改 → [全量重建]:
                          compute_dom_styles()  ← 遍历 DOM 树
                        → build_layout_tree()   ← 重建布局树
                        → layout()              ← 全量重新布局
                        → DisplayListBuilder    ← 重建绘制列表
                        → WgpuBackend::render() ← GPU 绘制
                        → swap_buffers
耗时: O(DOM大小 + 布局树大小 + 绘制命令数)
```

### Blink 的数据流

```
浏览器进程输入事件 → IPC → 渲染进程:
  EventHandler::HandleMouseEvent()
    → HitTest 定位目标元素
    → EventDispatcher::Dispatch() 在 EventPath 上传播
    → JS 回调修改 DOM
    → Node::SetNeedsStyleRecalc()    ← 仅标记脏节点
    → LayoutObject::SetNeedsLayout() ← 仅标记脏节点

BeginMainFrame:
  UpdateStyle()    ← 增量重算（仅遍历脏子树）
  UpdateLayout()   ← 增量重排（仅 LayoutNG dirty 节点）
  PrePaint         ← 更新 PropertyTrees
  Paint            ← 仅绘制变更的 DisplayItem
  Commit           ← 提交到合成器线程

合成器线程:
  Raster (tile worker pool) → Activate → Draw → CompositorFrame
  → IPC → viz 进程 → Display

耗时: O(脏节点数 + 脏绘制区域)，通常远小于 O(总大小)
```

## 关键架构差异总结

| 维度 | 我们 | Blink |
|------|------|-------|
| **语言** | Rust (安全 + 所有权) | C++ (灵活性 + 成熟生态) |
| **JS 执行** | 编译时翻译为 Rust | 运行时 V8 JIT 编译 |
| **线程模型** | 单线程同步 | 主线程 + 合成器线程 + tile worker 池 + GPU 进程 |
| **样式更新** | 全量重建 HashMap | StyleRecalcChange 增量传播 + 分组 CoW |
| **布局更新** | 全量 relayout() | LayoutNG 脏标记 + Fragment 缓存 |
| **绘制组织** | 扁平枚举 List | PaintChunk + PropertyTrees 分层 |
| **合成** | 无，直接绘制到窗口 | cc::Layer 树 → CompositorFrame → viz 聚合 |
| **光栅化** | 单帧全量重绘 | 分 tile 光栅化 + 缓存 |
| **内存管理** | Rc+RefCell 引用计数 | Oilpan 追踪式 GC |
| **平台抽象** | wgpu (跨平台 GPU) | Skia (跨平台 2D) + ANGLE/Metal/Vulkan |

## 改进方向

基于此全局对比，我们的架构演进方向按优先级排序：

1. **增量更新（P0）** — 从全量重建改为脏标记+局部重算，这是最大的性能瓶颈
2. **样式系统重构（P0）** — HashMap 全量克隆 → 分组 CoW ComputedStyle
3. **绘制分层（P1）** — 引入 PaintChunk + PropertyTrees，支持变换/透明度免重建
4. **布局缓存（P1）** — LayoutNG 风格的 Fragment 缓存复用
5. **合成器线程（P2）** — wgpu render pass 异步化，减少主线程负载
