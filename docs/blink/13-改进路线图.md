# 13 — 改进路线图：P0/P1/P2 优先级汇总

## 问题/目标

前 12 篇文档从 DOM、样式、布局、绘制、渲染等维度逐层对比了我们与 Blink 的实现。每篇文档末尾都列出了短期/中期/长期改进建议。本文将所有这些建议汇总、去重、排序，形成一条可执行的优化路线图。

**核心目标重申**：本项目的目标不是做一个通用浏览器，而是**将 HTML+CSS+JS 编译为原生跨平台桌面应用**。这意味着：
- 多进程隔离、合成器线程等浏览器特有的安全/并发架构，优先级大幅降低
- CSS 特性覆盖度（渐变、动画、过渡、滤镜）和渲染质量（抗锯齿、字形）优先级大幅提升
- 编译工具链体验（增量编译、构建速度、二进制体积）同样是核心关注点
- 对比 Blink 的目的不是复刻 Blink，而是从成熟引擎中借鉴可迁移的优化思路

## 优先级定义

| 级别 | 含义 | 时间范围 | 风险 | 判断标准 |
|------|------|---------|------|---------|
| **P0** | 立即执行 | 1-2 周 | 低（局部改动） | 「不改就是明显瓶颈」— 性能/内存问题直接影响 demo 体验 |
| **P1** | 短期规划 | 1-2 月 | 中（需要一定重构） | 「改了让应用更接近原生」— 架构能力提升，CSS 特性更完整 |
| **P2** | 长期愿景 | 3-6 月+ | 高（架构级变更） | 「决定能否作为实际工具使用」— 完整的跨端应用开发体验 |

## P0 — 立即可做，高收益低成本

P0 的核心思路：**利用已有但未使用的设施，或做局部微调即获得显著收益**。

### P0-1: 连接 dirty 标记到渲染管线 ⭐ 最高优先级

**来源**：[08-布局-增量更新.md](08-布局-增量更新.md)

**现状**：`Node.dirty`、`LayoutBox.dirty`、`mark_dirty()`/`clear_dirty()` 已完整实现，但 `App::relayout()` 完全忽略它们，每次全量重建。

**方案**：
- `compute_dom_styles()` 遍历时跳过 `!is_dirty()` 且父样式未变的节点
- `build_layout_box()` 对非脏节点复用上次的 `LayoutBox.rect`
- 脏节点 < 总数 50% 时走增量路径，否则回退全量

**预期收益**：1000 节点页面，修改 1 个属性 → 耗时从 ~50ms 降至 ~5ms（10x 提升）

**改动范围**：[`crates/renderer/src/window.rs`](../../crates/renderer/src/window.rs) `relayout()` 方法

### P0-2: 惰性分配 events HashMap

**来源**：[03-DOM-事件系统.md](03-DOM-事件系统.md)

**现状**：每个 `ElementData` 初始化时分配空 `HashMap`（~48 bytes），99% 的元素无事件监听器。

**方案**：
```rust
// before
pub(crate) events: HashMap<String, Vec<EventListener>>,
// after
pub(crate) events: Option<HashMap<String, Vec<EventListener>>>,
```

**预期收益**：1000 元素页面节省 ~48KB（仅空 HashMap），首次 `addEventListener` 时才分配

**改动范围**：[`crates/dom/src/element.rs`](../../crates/dom/src/element.rs) `ElementData` 结构体

### P0-3: 预编译选择器 + 规则按类型索引

**来源**：[04-CSS-选择器匹配.md](04-CSS-选择器匹配.md)

**现状**：`parse_selector_parts()` 对每个元素×每条规则都重新解析选择器字符串，外循环 O(E×R×S)。

**方案**：
1. 在 `StyleSheet` 加载时预解析选择器为 `Vec<SelectorPart>`，缓存于规则中
2. 建立三个索引：
   ```rust
   struct RuleIndex {
       id_rules: HashMap<String, Vec<usize>>,    // #id → 规则索引
       class_rules: HashMap<String, Vec<usize>>, // .class → 规则索引
       tag_rules: HashMap<String, Vec<usize>>,   // tag → 规则索引
       universal_rules: Vec<usize>,              // * → 规则索引
   }
   ```
3. 先查索引，再做 RTL 方向的最右段剪枝

**预期收益**：500 元素×50 规则的匹配从 25000 次降至 ~500 次（50x 减少）

**改动范围**：[`crates/style/src/selector.rs`](../../crates/style/src/selector.rs)，[`crates/style/src/stylesheet.rs`](../../crates/style/src/stylesheet.rs)

### P0-4: 样式计算结果缓存

**来源**：[05-CSS-级联与计算样式.md](05-CSS-级联与计算样式.md)

**现状**：每次 `compute_dom_styles()` 全量重新级联所有元素，即使样式规则、DOM 结构、元素属性均未变更。

**方案**：

**改动范围**：[`crates/style/src/cascade.rs`](../../crates/style/src/cascade.rs)，[`crates/renderer/src/window.rs`](../../crates/renderer/src/window.rs)

### P0-5: 视口剔除 + 边框解析缓存

**来源**：[09-绘制-DisplayList 构建.md](09-绘制-DisplayList 构建.md)，[10-绘制-合成与优化.md](10-绘制-合成与优化.md)

**方案**：
1. `process_node()` 入口处检查 `node.rect` 与视口是否相交，不相交则跳过
2. `extract_border()` 结果缓存在 `LayoutBox` 上（或从 ComputedStyle 实时计算）

**预期收益**：视口外 50% 节点跳过 → DisplayList 构建耗时减半

**改动范围**：[`crates/render_tree/src/builder.rs`](../../crates/render_tree/src/builder.rs)

### P0-6: DFS 自然顺序消除全局排序

**来源**：[09-绘制-DisplayList 构建.md](09-绘制-DisplayList 构建.md)

**现状**：`DisplayListBuilder::build()` 末尾对所有命令执行 `sort_by_z_order()`（O(N log N)）。

**方案**：DFS 遍历时按背景→子元素→前景的顺序将命令插入不同 bucket，遍历完成后拼接：

```rust
// 7 个 bucket，按层顺序
buckets[0] // z < 0
buckets[1] // z = 0, box-shadow
buckets[2] // z = 0, fill
buckets[3] // z = 0, border
buckets[4] // z = 0, text
buckets[5] // z > 0
buckets[6] // overflow/stacking
```

**预期收益**：消除 O(N log N) 排序，降至 O(N) bucket 插入后 concat

**改动范围**：[`crates/render_tree/src/builder.rs`](../../crates/render_tree/src/builder.rs)

### P0-7: SRGB 批量预计算 + computed_style 实时取值

**来源**：[05-CSS-级联与计算样式.md](05-CSS-级联与计算样式.md)，[10-绘制-合成与优化.md](10-绘制-合成与优化.md)，[06-布局-盒模型与树结构.md](06-布局-盒模型与树结构.md)

**方案**：
1. CSS 颜色解析时将 sRGB 值转为线性并存储，避免每次渲染转换
2. `padding()`/`margin()`/`border_width()` 方法改为从 `computed_style` 实时计算，消除 `LayoutBox` 上的 `EdgeSizes` 冗余

**预期收益**：减少 LayoutBox 内存占用~48 bytes，消除渲染时转换开销

**改动范围**：[`crates/style/src/values.rs`](../../crates/style/src/values.rs)，[`crates/layout/src/layout_box.rs`](../../crates/layout/src/layout_box.rs)

### P0-8: 属性名常量 + avoid clone in fire_listeners

**来源**：[05-CSS-级联与计算样式.md](05-CSS-级联与计算样式.md)，[03-DOM-事件系统.md](03-DOM-事件系统.md)

**方案**：
1. CSS 属性名 `HashMap<String, CSSValue>` → key 用 `&'static str` 或枚举
2. `fire_listeners_on_node` 中先收集 ID 再查找 → 直接在迭代中处理

**预期收益**：减少字符串分配，简化事件派发

### P0 汇总

| 编号 | 改进项 | 预期收益 | 改动文件数 | 风险 |
|------|--------|---------|-----------|------|
| P0-1 | 连接 dirty 标记 | **10x 渲染性能** | 1 | 低 |
| P0-2 | 惰性 events HashMap | ~48KB/千元素 | 1 | 低 |
| P0-3 | 选择器预编译+索引 | **50x 选择器匹配** | 2 | 低 |
| P0-4 | 样式结果缓存 | 跳过非脏元素级联 | 2 | 低 |
| P0-5 | 视口剔除 | ~2x DisplayList 构建 | 1 | 低 |
| P0-6 | DFS bucket 消除排序 | O(N log N)→O(N) | 1 | 低 |
| P0-7 | SRGB 预计算+实时取值 | ~48B/LayoutBox | 2 | 低 |
| P0-8 | 属性名常量 | 减少字符串分配 | 2 | 低 |

## P1 — 中期架构优化

P1 不再是小修小补，而是结构性改进，需要合理的缓存策略和分层设计。

### P1-1: CoW ComputedStyle（写时复制样式）

**来源**：[05-CSS-级联与计算样式.md](05-CSS-级联与计算样式.md)

**方案**：
```rust
use std::sync::Arc;

#[derive(Clone)]
struct ComputedStyle {
    box_model: Arc<BoxModelStyle>,       // width, height, padding, margin, border
    typography: Arc<TypographyStyle>,    // font-size, color, line-height, text-align
    visual: Arc<VisualStyle>,            // background, opacity, visibility, transform
    layout: Arc<LayoutStyle>,            // display, position, flex-direction, grid-*
    rare: Option<Arc<RareStyle>>,        // filter, backdrop-filter, mix-blend-mode
}
// 继承时全共享 Arc → 0 拷贝
// 修改时 Arc::make_mut → 仅复制被修改的组（~100 bytes）
```

**预期收益**：父子元素样式继承从 O(40 属性) 拷贝降至 O(1) Arc inc

**改动范围**：[`crates/style/src/cascade.rs`](../../crates/style/src/cascade.rs)，新增 `crates/style/src/computed.rs`

### P1-2: PropertyTrees — 变换/裁剪/透明度分离

**来源**：[10-绘制-合成与优化.md](10-绘制-合成与优化.md)

**方案**：
```rust
struct PropertyTrees {
    transforms: Vec<TransformNode>,  // 变换树
    clips: Vec<ClipNode>,            // 裁剪树
    effects: Vec<EffectNode>,        // 效果树（opacity, filter）
}

struct PaintCommand {
    cmd_type: PaintCommandType,
    geometry: Geometry,
    transform_id: u32,  // ★ 指向 PropertyTrees 节点
    clip_id: u32,       // ★ 指向 PropertyTrees 节点
    effect_id: u32,     // ★ 指向 PropertyTrees 节点
}
// 修改 opacity 只需更新 EffectNode → 0 个 PaintCommand 受影响
```

**预期收益**：CSS transform/opacity 变更时 DisplayList 零重建

**改动范围**：新增 `crates/render_tree/src/property_trees.rs`，修改 `builder.rs`、`command.rs`

### P1-3: 增量布局 — 实现 partial_layout

**来源**：[08-布局-增量更新.md](08-布局-增量更新.md)

**方案**：
```rust
impl LayoutEngine {
    fn partial_layout(&mut self, root: &mut LayoutBox, dirty_nodes: &[Rc<RefCell<Node>>], viewport: Size<f32>) {
        for dirty in dirty_nodes {
            if let Some(layout_node) = root.find_layout_node_mut(dirty) {
                // 1. 仅重算该节点的尺寸
                self.calculate_sizes(layout_node, viewport);
                // 2. 重排其子节点
                self.layout_children(layout_node);
                // 3. 向下传播位置变更（兄弟节点 y 偏移）
                self.propagate_position_changes(layout_node);
            }
        }
    }
}
```

**预期收益**：单节点修改从 O(N) 降至 O(子树大小)

**改动范围**：[`crates/layout/src/lib.rs`](../../crates/layout/src/lib.rs) — 扩展现有的 `partial_layout()` stub

## P2 — 长期：完整的跨端应用开发体验

P2 的目标不再是追赶 Blink 的浏览器能力，而是让 HTML+CSS+JS → 原生应用的这条编译路线**真正可被开发者使用**。核心围绕三个方向：**CSS 特性完整度**、**渲染品质**、**编译工具链体验**。

### P2-1: 完整 CSS 图形特性

**来源**：[09-绘制-DisplayList 构建.md](09-绘制-DisplayList 构建.md)，[11-渲染-GPU管线.md](11-渲染-GPU管线.md)

当前仅支持纯色填充 + 纯色边框 + SDF 圆角 + BoxShadow。实际应用至少需要：

| 特性 | 当前状态 | 目标 |
|------|---------|------|
| `linear-gradient` / `radial-gradient` | ❌ | WGSL shader 实现，作为 FillRect 的变体 |
| `conic-gradient` | ❌ | 同上 |
| `box-shadow` 多层 | 仅单层 | 支持逗号分隔的多层阴影 |
| `text-shadow` | ❌ | Text 命令增加阴影参数 |
| `filter: blur()` | ❌ | 离屏渲染 + WGSL blur pass |
| `filter: drop-shadow()` | ❌ | 同上 |
| `backdrop-filter` | ❌ | 需要先采样背景纹理 |
| `mix-blend-mode` | ❌ | wgpu 原生支持 blend state |
| `opacity` (元素级) | 仅 Clip/Opacity stub | 完整实现 per-element opacity |
| `transform` (2D) | 仅解析 | rotate/scale/translate/skew → 实例数据矩阵 |
| `border-image` | ❌ | 与 image 渲染共用纹理采样逻辑 |
| `clip-path` | ❌ | 基本形状 (circle/ellipse/polygon) → SDF shader |

**预期收益**：应用开发者可以用 CSS 实现渐变背景、毛玻璃效果、阴影层级、动效变换

**改动范围**：`crates/renderer/src/wgpu_backend.rs`（新增 shader），`crates/render_tree/src/command.rs`（新增命令变体），`crates/style/src/values.rs`（解析渐变语法）

### P2-2: 图片渲染

**来源**：[10-绘制-合成与优化.md](10-绘制-合成与优化.md)

**当前状态**：`PaintCommand::Image` 枚举变体已定义，`TextureAtlas` stub 已存在，但无实际图片解码和渲染逻辑。

**方案**：
```rust
// 解码 → 纹理图集 → GPU 采样 → 渲染
use image crate → wgpu::Texture → WGSL textureSample
```

1. 引入 `image` crate（纯 Rust，支持 PNG/JPEG/WebP/GIF/BMP）
2. `background-image: url(...)` 解析 + 图片加载
3. `<img>` 元素的固有尺寸 + 布局 + 绘制
4. `object-fit` (contain/cover/fill/none/scale-down) 支持
5. 图片缓存（避免重复解码）

**预期收益**：支持背景图、图标、产品图等所有应用必需的视觉元素

**改动范围**：`crates/renderer/src/texture_atlas.rs`（实现），`crates/render_tree/src/command.rs`，`crates/renderer/src/wgpu_backend.rs`（新增 image shader pipeline）

### P2-3: 完整动画与过渡引擎

**来源**：[05-CSS-级联与计算样式.md](05-CSS-级联与计算样式.md)

**当前状态**：`Animations` / `Transitions` 模块有基本的 keyframe 注册和属性插值，但与渲染管线未连通。`animation` / `transition` 的 CSS 简写属性未解析。

**方案**：
1. CSS `transition` 属性解析：`property duration easing delay`
2. CSS `animation` 属性解析：`name duration easing delay iteration-count direction fill-mode`
3. 每一帧 tick 时：
   - 遍历活跃的 transition/animation
   - 根据 elapsed/total 计算进度
   - 对目标属性插值 → 写入 ComputedStyle
   - 触发布局/绘制更新
4. 缓动函数完整实现（ease/ease-in/ease-out/ease-in-out/linear/cubic-bezier/steps）

**预期收益**：hover 过渡、淡入淡出、位移动画——应用交互体验的核心

**改动范围**：`crates/style/src/animations.rs`，`crates/style/src/transitions.rs`，`crates/style/src/values.rs`（简写属性解析），`crates/renderer/src/window.rs`（帧 tick 驱动）

### P2-4: 高质量文本渲染

**来源**：[11-渲染-GPU管线.md](11-渲染-GPU管线.md)，[07-布局-核心算法.md](07-布局-核心算法.md)

**当前状态**：`ab_glyph` 光栅化字形到纹理图集，rustybuzz (HarfBuzz) 用于文本塑形。但存在几个问题：
- 字形图集每次渲染重建
- 无 subpixel 定位
- 无文本抗锯齿选项
- 仅支持系统字体，无 Web 字体加载

**方案**：
1. **字形图集持久化**（P1-7 已完成时跳过）
2. **Subpixel glyph positioning**：字形 x 偏移使用 `f32` 而非四舍五入
3. **Web 字体**：支持 `@font-face { src: url(...) }` 加载自定义字体
4. **文本换行**：`word-break`、`overflow-wrap`、`white-space` 完整实现
5. **`text-overflow: ellipsis`**：溢出文本显示省略号

**预期收益**：文本渲染品质达到原生应用水平

**改动范围**：`crates/renderer/src/text_renderer.rs`，`crates/layout/src/text.rs`

### P2-5: 编译器工具链体验

**来源**：[01-总览-架构对比.md](01-总览-架构对比.md)

这是"HTML+CSS+JS → 原生应用"路线中 Blink 完全没有的部分，也是本项目的核心差异。

**方案**：

1. **增量编译**：
   - 当前每次 build 重新编译全部源文件
   - 检测文件修改时间 → 仅重新编译变更的文件
   - 配合 P0-1 dirty 标记 → 只重新生成变更节点的代码

2. **Watch 模式**：
   ```bash
   cargo run -p toolchain -- watch ./app/ --target ./generated/
   # 文件变更 → 自动重新编译 → 自动 cargo build
   ```

3. **HMR（热模块替换）**：
   - 样式变更 → 只更新 ComputedStyle → 不重建 DOM
   - 文本变更 → 只更新 TextCommand → 不重建布局

4. **错误定位**：
   - CSS 解析错误 → 输出文件名:行号:列号
   - JS 模式识别失败 → 指出不支持的 API 及替代方案
   - HTML 解析错误 → 指出嵌套问题

5. **二进制体积优化**：
   - `opt-level = "s"` / LTO / strip
   - wgpu shader 预编译为二进制而非嵌入 WGSL 源码
   - 移除未使用的 CSS 属性解析代码（feature flag 裁剪）

**预期收益**：从「能跑 demo」到「开发者愿意使用」的关键跨越

**改动范围**：`crates/toolchain/src/lib.rs`，`crates/toolchain/src/js.rs`，新增 watch 模式

### P2-6: 组件与 UI 框架兼容

**来源**：实际应用场景

实际应用不会手写所有 CSS——开发者会使用 CSS 框架或组件库。

**方案**：

1. **CSS 框架兼容性测试**：
   - Tailwind CSS 常用 class → 验证选择器 + 属性解析覆盖率
   - Bootstrap 按钮/卡片/导航 → 验证 Flexbox + 常用属性
   - 整理「已支持 / 不支持」清单

2. **CSS 自定义属性（`var()`）完整支持**：
   - 当前 `CustomProperties` 模块有基础设施但未完全集成
   - Tailwind 等框架重度依赖 CSS variables

3. **`@media` 查询扩展**：
   - 当前仅支持 `width`，需扩展 `prefers-color-scheme`、`prefers-reduced-motion`
   - 桌面应用的「暗色模式」依赖此特性

4. **`@keyframes` 完整语法**：
   - `from/to` 简写
   - 百分比关键帧
   - 多属性同一动画

**预期收益**：可直接使用 Tailwind CSS / Bootstrap 写样式，大幅降低开发者迁移成本

**改动范围**：`crates/style/src/media.rs`，`crates/style/src/custom_props.rs`，`crates/toolchain/src/css.rs`

### P2-7: 跨平台能力完善

**来源**：项目定位

**方案**：

1. **wasm 目标验证**：wgpu 已支持 WebGPU 后端，验证 `cargo build --target wasm32-unknown-unknown` 可用
2. **移动端适配**：winit 支持 iOS/Android，验证触摸事件和屏幕旋转

4. **系统集成**：系统托盘、通知、文件对话框（通过 winit 扩展或独立 crate）

**预期收益**：真正的「一次编写，多端运行」

### P2-8: 测试与质量保障

**来源**：[07-布局-核心算法.md](07-布局-核心算法.md)

**方案**：

1. **CSS 属性覆盖率报告**：自动生成「已解析 / 不支持」属性清单
2. **可视化回归测试**：golden image diff 对比渲染输出

| 移除项 | 原因 |
|--------|------|

## 建议执行顺序

```
第1-2周（P0 全部 — 性能基准）:
  P0-1 dirty 标记连接 ──────── 最大收益，立即见效
  P0-2 惰性 events HashMap ─── 几乎零成本
  P0-3 选择器预编译+索引 ──── 样式计算加速
  P0-4 样式结果缓存 ────────── 配合 P0-1
  P0-5 视口剔除 ─────────────── 配合 P0-1
  P0-6 DFS bucket ──────────── 无风险优化
  P0-7 SRGB+实时取值 ──────── 持续清理
  P0-8 属性名常量 ──────────── 持续清理

第3-8周（P1 — 架构能力）:
  第3周: P1-5 Bloom Filter + P1-7 字形缓存 + P1-9 增量实例缓冲
  第4-5周: P1-4 StyleDifference 检测 + P1-3 增量布局
  第6-7周: P1-1 CoW ComputedStyle
  第8周: P1-8 ConstraintSpace + P1-6 PaintChunk + P1-2 PropertyTrees

第3-6月（P2 — 跨端应用体验）:
  第3月: P2-1 渐变+滤镜+变换（CSS 图形特性）
  第4月: P2-2 图片渲染
  第5月: P2-3 动画/过渡引擎
  第6月: P2-4 高质量文本 + P2-5 编译器工具链
  持续: P2-6 UI 框架兼容 + P2-7 跨平台完善 + P2-8 测试
```

## 关键决策点

| 决策点 | 触发条件 | 选项 A | 选项 B |
|--------|---------|--------|--------|
| **渐变实现** | P2-1 启动时 | WGSL shader（保持 wgpu 一致性） | tiny-skia CPU 光栅化（更简单、AA 免费） |
| **动画引擎** | P2-3 启动时 | 自研（CSS 属性插值，与 CoW ComputedStyle 深度集成） | 引入现有动画库（更成熟但耦合度高） |
| **编译器升级** | P2-5 时 | 继续完善手写解析器（零依赖，小体积） | 引入 swc/cssparser（更健壮，但增大二进制） |
| **Web 字体** | P2-4 时 | fontdb（已集成）+ 网络下载 | 系统字体 only（简化但限制开发者） |
| **HMR 策略** | P2-5 编译体验 | 仅样式热替换（风险最低，最常见场景） | 完整 HMR（样式+布局+DOM，复杂度高） |
