# Rust 浏览器引擎 — 可行性报告

## 一、项目概述

### 项目定位
用 Rust 构建一套符合 W3C 标准的浏览器渲染引擎。核心创新在于**无嵌入式 JS 引擎** —— 通过构建期的 JS→Rust 编译将标准 Web 内容转换为 Rust 原生代码执行。

### 核心挑战
这不是"浏览器引擎的子集"，而是完整的浏览器标准覆盖（HTML + CSS + DOM + ES + Web API）。工程量取决于标准覆盖的深度，但分期策略可使每个阶段都可交付。

## 三、关键技术难点分析

### 3.1 JS→Rust 编译器（风险等级：🔴🔴 最高）

这是整个项目最核心也最困难的技术挑战。

**原理**：
```
JavaScript (动态类型, 原型继承, 运行时)
    ↓ swc parser
JS AST
    ↓ 语义分析 + 类型推断
IR (中间表示)
    ↓ Rust 代码生成
Rust 代码 (静态类型, trait 继承, 编译时)
```

**关键难点**：

| JS 特性 | 难度 | Rust 映射方案 | 说明 |
|---------|------|---------------|------|
| let/const/var | ★★ | let/mut | 作用域规则类似 |
| 函数 + 闭包 | ★★★ | fn + closure | 捕获方式映射 (move/ref) |
| 对象字面量 | ★★★ | HashMap / struct | 有结构字段→struct, 无结构→HashMap |
| 原型链 | ★★★★★ | trait + 组合 | 原型链最复杂，需要运行时缓存 |
| this 绑定 | ★★★★ | 显式 this 参数 | JS 的 this 动态性需特殊处理 |
| Promise/async | ★★★★ | Future + async | 概念天然对齐 |
| 类型强制转换 | ★★★★ | 显式转换调用 | 需要插入类型转换代码 |
| Proxy | ★★★★★ | 运行时反射 | 基本不可静态编译，需 fallback |
| eval | ❌ | 不支持 | 声明不支持 eval |
| 动态属性访问 | ★★★★ | Index trait | obj[key] → std::ops::Index |

**缓解策略**：
1. 分阶段实现：Phase 0 模式识别 → Phase 1 swc 完整 AST 编译
2. 不支持的特性（eval、with）编译时报错

### 3.2 CSS 引擎（风险等级：🟡 中）

**难点**：
- 500+ 个标准属性，每个的计算逻辑不同
- 简写属性展开（background → ~8 个子属性）
- 特异性计算、级联顺序、!important 规则
- 动画/过渡的时间线 + 插值

**工作量估算**：
```
属性定义 (properties.toml)    ~3000 行
cssparser 集成                ~500 行
选择器匹配                     ~1000 行
级联引擎                      ~2000 行
值计算                        ~3000 行
动画/过渡引擎                  ~3000 行
─────────────────────────────────
总计                          ~12500 行
```

**缓解策略**：
- 数据驱动：TOML 定义属性元数据 + 代码生成
- 一期覆盖 80 个属性（日常使用 90%+）
- 参考 Servo 的 `style` crate 设计

### 3.3 布局引擎（风险等级：🟡 中）

| 布局模式 | 行数估算 | 风险 | 策略 |
|----------|---------|------|------|
| Flexbox | ~500 (taffy 集成) | 🟢 | 直接用 taffy |
| Grid | ~500 (taffy 集成) | 🟢 | 直接用 taffy |
| Block | ~1500 | 🟡 | 自研，算法经典 |
| Inline | ~2500 | 🟡 | 自研，行框构建复杂 |
| Table | ~3000 | 🔴 | 算法复杂，优先级低 |
| Float | ~2000 | 🟡 | 自研，经典算法 |
| Positioned | ~1000 | 🟢 | 相对简单 |
| Text Layout | ~3000 | 🔴 | rustybuzz + 换行 + 字体回退 |

**Table 布局的复杂性**：
Table 布局是 CSS 中最复杂的布局模式之一，涉及：
- 列宽分配算法（固定/自动/百分比）
- 跨行/跨列 (rowspan/colspan)
- 双 pass 布局（先算列宽再布局内容）
- 边框合并 (border-collapse)

➡ **建议 Table 放在 Phase 2，非核心路径**

### 3.4 wgpu 渲染管线（风险等级：🟡 中）

**难点**：
1. 渲染合批：减少 Draw Call 是关键优化
2. 文本渲染：SDF (Signed Distance Field) 或 bitmap 缓存
3. 滚动裁剪：scissor rect + clip 链
4. 层合成 (Compositing)：将页面分层独立渲染

**wgpu 平台支持矩阵**：

| 平台 | 后端 | wgpu 支持 | Phase |
|------|------|-----------|-------|
| Windows | DX12 / Vulkan | ✅ 完整 | P0 |
| macOS | Metal | ✅ 完整 | P0 |
| Linux | Vulkan | ✅ 完整 | P0 |
| Web | WebGPU / WebGL | ✅ 支持 | P2 |
| iOS | Metal | ✅ 支持 | P3 |
| Android | Vulkan | ✅ 支持 | P3 |

## 六、结论

### 可行，但需要分阶段长期投入

| 维度 | 结论 |
|------|------|
| 技术可行性 | ✅ **核心路径可行** — DOM/CSS/Layout 均有成熟的 Rust 生态基础 |
| 最大风险 | 🔴 **JS→Rust 编译器** — 是最大创新点，也是最大技术挑战 |
| 初期策略 | 🟢 **Phase 0 低风险** — 运行时可用 + 基础版 web2rust 编译器 |
| 长期策略 | 🟡 **分期逐步扩展** — 每期有明确交付物，控制范围蔓延 |

### 关键建议

1. **Phase 0 先走通全链路**：HTML+CSS+JS 源文件 → web2rust 编译 → 运行时管线 → GPU 渲染
2. **JS 编译器分步走**：Phase 0 只做模式识别，Phase 1 升级为 swc 完整编译
3. **CSS 属性数据驱动**：用 TOML 定义 + 代码生成，避免手写每个属性
4. **文本渲染尽早验证**：Phase 0 就接入 rustybuzz，暴露核心难点
5. **Table 和 Canvas 押后**：这两个是最复杂的子系统，放到 Phase 2+
